# 사건 데이터 스키마 및 사용 가이드

본 문서는 `case_interrogation_01.json`과 동일한 구조의 사건 데이터가
게임 심문 시스템에서 어떻게 사용되는지 설명한다.

---

## 1. 기본 개념

- 사건 데이터는 **한 판의 심문 동안 절대 변하지 않는 고정 정보**이다.
- 심문 중 AI는 새로운 사실, 증거, 모순을 생성하지 않는다.
- 모든 질문·압박·자백 판단은 이 JSON에 포함된 정보만을 근거로 한다.

---

## 2. 필드별 설명

### case_id
- 사건 고유 ID
- UE와 서버 간 사건 식별용
- 한 판 동안 동일하게 유지

---

### overview
사건의 기본 배경 정보

- time: 사건 발생 시각
- place: 사건 장소
- type: 사건 유형

📌 사용처
- 사건 브리핑 UI
- 플레이어가 시간·장소 관련 질문을 할 수 있는 근거

---

### motive
범행 동기

📌 사용처
- 후반부 심문에서 “왜 그랬는지” 추궁할 때 사용
- 자백 대사 생성 시 감정적 설득 근거

---

### crime_flow
실제 범행의 사실 관계 요약 (정답)

📌 사용처
- AI 내부 참고용
- 플레이어에게 직접 노출되지 않음
- 자백 트리거 이후의 인정 발화 생성에 사용

---

### suspect
용의자 기본 정보

- name
- age
- job
- relation

📌 사용처
- 브리핑 UI
- 용의자 소개
- 초반 신뢰도 형성

---

### false_statement
용의자가 **초기 단계에서 반복적으로 주장하는 거짓 진술**

📌 사용처
- 압박도가 낮을 때 AI 응답의 기준선
- 초반에는 이 진술을 유지하며 버팀

---

### evidences (증거 목록)

각 증거는 심문에서 **압박을 증가시키는 트리거** 역할을 한다.

필드:
- id: 내부 식별자 (압박 계산용 키워드)
- name: UI 표시용 이름
- description: NPC가 반응할 수 있는 설명

📌 사용처
- 플레이어 질문에 증거 관련 단어가 포함되면 pressure_delta 증가
- AI가 “흔들리는 반응”을 보이기 시작하는 근거

---

### contradictions (모순점)

거짓 진술과 증거 사이의 충돌 지점

필드:
- id: 모순 식별자
- description: 어떤 점이 모순인지 설명
- related_evidence: 연결된 증거 id 목록

📌 사용처
- 모순을 정확히 찌르면 pressure_delta가 크게 증가
- confession_probability 상승의 핵심 요인

---

## 3. 압박 / 자백 계산 개념 (2주차 기준)

- 증거 언급: pressure_delta +0.2
- 모순 지적: pressure_delta +0.35
- 누적 압박 + 질문 횟수에 따라 confession_probability 증가
- confession_probability ≥ 0.85 → 자백 트리거

📌 수치는 2주차 기준 임시값이며,
3주차에 난이도 조절을 위해 조정 가능

---

## 4. 중요한 규칙

- 필드명 변경 금지
- 필드 삭제 금지
- 새로운 필드 추가는 3주차 이후 논의

본 스키마는 **UE ↔ 서버 간 계약 문서**로 취급한다.
